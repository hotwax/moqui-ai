# Agent & Tool Registry Keystone — Implementation Plan (TDD)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Re-model `AiTool` and `AiAgent` as records with a **stable opaque id** (`toolId` / `agentId`) and **editable, human-friendly names** (`toolName = verb_noun`, `agentName`), make the **DB the source of truth** for the tool catalog (replacing the `ai/*.tools.xml` boot scan), and add the **single authoring gate** (`store#AiTool` / `store#AiAgent`) that validates the backing service, derives `effect`, applies a non-overridable denylist, and enforces unique names. The agentic run loop, all providers, all tests, and the NotNaked demo move to the id model. This is the keystone the future builder agent sits on.

**Architecture:** `toolId`/`agentId` are sequenced opaque PKs; `toolName`/`agentName` are unique non-PK attributes. The grant (`AiAgentTool`), failover chain (`AiAgentModel`), runs (`AiAgentRun`), and conversations (`AiConversation`) re-key from names to ids; `AiAgentRun`/`AiConversation` keep `agentName` and `AiToolCall` keeps `toolName` as **denormalized display snapshots** so history reads correctly after a rename. `AiToolFactory` holds an in-memory catalog **built from `AiTool` rows** (keyed by `toolId`, also indexed by `toolName`), refreshed on demand; the JSON schema for each tool is generated by `ToolSchemaBuilder` from the *live* service at catalog-build time (never stored). `store#AiTool` is the only path that turns a service into a tool: it validates the service exists (fail-loud via `ToolSchemaBuilder`), derives `effect` from the verb, applies the `AiToolDenylist` floor, sets role/effect-based defaults, derives + uniqueness-checks `toolName`. `store#AiAgent` sequences `agentId` and enforces unique `agentName`. The `AgentRunner` loads the agent by `agentId` (resolvable by `agentName`), maps grants → `AiTool` rows → `request.tools` (wire name = `toolName`, dispatch via `serviceName`); the provider `sanitizeName`/`remapToolNames` round-trip collapses to a thin no-op safety net because `verb_noun` is wire-safe by construction.

**Tech Stack:** Groovy 3 / JDK 11, Moqui (HotWax fork), Spock + JUnit Platform Suite, H2 dev DB.

## Implements

The **Agent & Tool Registry** design spec: `docs/specs/2026-06-05-agent-tool-registry-design.md` (decisions **D1–D5**, data model §4, naming/derivation §5, the one-gate validation/safety/schema §6, data flow §7, impact §8, testing §9). **Supersedes for tools/agents** the in-memory catalog built from `ai/*.tools.xml` by `DefinitionLoader`.

## Locked decisions (from the spec)

- **D1 — Stable opaque identity, editable names.** `toolId`/`agentId` opaque stable PKs; `verb`/`noun`/`toolName`/`description` and `agentName`/`description` mutable. `serviceName` is a plain attribute, never a key.
- **D2 — Two roles.** Capability Curator authors tools (`store#AiTool`); Agent Composer assembles agents (`store#AiAgent` + grants) and can never expose a raw service.
- **D3 — DB is the source of truth; seeded as standard data.** Tools/agents are entity records in `data/*.xml`; runtime edits are DB writes. Replaces the `ai/*.tools.xml` scan.
- **D4 — Safety: derive + floor.** `effect` (READ_ONLY | MUTATING) is derived from the service verb; read-only auto-exposable, mutating Curator-only + `requiresApproval=Y` default. A non-overridable `AiToolDenylist` forces `exposable=N`.
- **D5 — Naming: guidance, not enum.** `toolName = verb_noun`, derived, unique, wire-safe by construction.

## Plan-level reconciliations (confirm-at-implementation calls, locked here)

These resolve the spec's "confirm at planning" open items against the UDM practices guide (`/Users/anilpatel/maarg-sd/docs/udm-domain-object-practices.md`) and the existing code:

1. **`effect` archetype → Enumeration, field name `effectEnumId`.** UDM §1.2: a fixed, code-referenced categorical value with no extra structured fields is an **Enumeration**, field suffix `EnumId`. The spec's §4.1 field `effect` (type `id`, values `AI_TOOL_READ_ONLY|AI_TOOL_MUTATING`) becomes **`effectEnumId`** (type `id`) backed by `moqui.basic.Enumeration` records with `enumTypeId="AiToolEffect"`. (Statuses stay `StatusItem` per UDM §1.5; effect is not a lifecycle.)
2. **`AiTool` / `AiAgent` are Master archetype** (UDM §1.1): single synthetic PK (`toolId` / `agentId`), framework-sequenced for user-created rows (`sequencedIdPrimary`), explicit stable ids in seed data. `AiAgentTool` / `AiAgentModel` are Join archetype (two PKs); they re-key to `(agentId, toolId)` / `(agentId, priority)`.
3. **`agentName` / `toolName` uniqueness** via an entity-level `<index ... unique="true">` (confirmed syntax in `runtime/component/maarg-util/entity/ExecutionAgentEntities.xml`). H2 enforces it; a duplicate `create` throws, which `store#AiTool`/`store#AiAgent` pre-empt with an explicit find-and-fail for a clean message.
4. **Denormalized snapshots stay as plain fields:** `AiAgentRun.agentName`, `AiConversation` keeps no agentName once re-keyed *except* `AiAgentRun.agentName` (needed by `CostServices.get#AiSpend`, which filters runs by `agentName`) — see Task 7. `AiToolCall.toolName` already exists and stays the display snapshot; we add `toolId` + (already-present) `serviceName`.
5. **Catalog key.** `AiToolFactory` indexes the catalog by **`toolId`** (primary) and by **`toolName`** (for wire-name → tool resolution when a provider returns a tool call). The run loop sends `toolName` on the wire and resolves the returned name back to the tool via the `toolName` index. `getTool(...)` lookups in `AgentRunner` switch from serviceName-keyed to id/name-keyed (see Task 5/7).
6. **Pre-release, no migration.** Dev tables are dropped/recreated. Moqui auto-adds new columns on boot; PK shape changes and a `type` change on a PK field require dropping the affected tables first (Task 0). No data-migration code.

```groovy
package org.moqui.ai

import org.moqui.context.ExecutionContextFactory
import org.moqui.entity.EntityValue

/** Builds the in-memory tool catalog from AiTool rows (DB is the source of truth — spec D3).
 *  Only ACTIVE + exposable tools are grant-eligible. The JSON schema for each tool is generated
 *  on demand from the LIVE service via ToolSchemaBuilder (never stored — spec §6). Returns a NEW
 *  map keyed by toolId; the factory swaps it atomically and also indexes it by toolName. */
class DefinitionLoader {
    static Map<String, Map> loadCatalog(ExecutionContextFactory ecf) {
        Map<String, Map> catalog = [:]
        boolean disabled = ecf.getExecutionContext().artifactExecution.disableAuthz()
        try {
            for (EntityValue t in ecf.getExecutionContext().entity.find("moqui.ai.AiTool")
                    .condition("statusId", "AI_TOOL_ACTIVE").condition("exposable", "Y").list()) {
                String serviceName = t.serviceName as String
                Map schema
                try {
                    schema = ToolSchemaBuilder.build(ecf, serviceName)   // fail-loud per tool
                } catch (Exception e) {
                    // a seeded tool whose service was removed should not break the whole catalog at boot;
                    // log and skip (the authoring gate validates at write time — this is a defensive read).
                    org.slf4j.LoggerFactory.getLogger(DefinitionLoader).warn(
                        "AiTool ${t.toolId} (${t.toolName}) references missing service ${serviceName}; skipping")
                    continue
                }
                catalog.put(t.toolId as String, [toolId: t.toolId, toolName: t.toolName,
                    serviceName: serviceName, description: t.description,
                    requiresApproval: (t.requiresApproval == "Y"), effectEnumId: t.effectEnumId,
                    schema: schema])
            }
        } finally {
            if (disabled) ecf.getExecutionContext().artifactExecution.enableAuthz()
        }
        return catalog
    }
}
```

```xml
<?xml version="1.0" encoding="UTF-8"?>
<services xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
        xsi:noNamespaceSchemaLocation="http://moqui.org/xsd/service-definition-3.xsd">

    <!-- The single authoring gate for turning a Moqui service into a tool (spec D2/D4/§6).
         Curator-gated via ArtifactAuthz. Validates the service, derives effect, applies the
         denylist floor, sets effect-based defaults, derives + uniqueness-checks toolName. -->
    <service verb="store" noun="AiTool" authenticate="true">
        <in-parameters>
            <parameter name="toolId"><description>Omit to create (sequenced); set to update an existing tool by stable id.</description></parameter>
            <parameter name="verb" required="true"/>
            <parameter name="noun" required="true"/>
            <parameter name="serviceName" required="true"/>
            <parameter name="description"/>
            <parameter name="exposable"><description>Requested exposability; floored to N by the denylist and by effect=MUTATING default.</description></parameter>
            <parameter name="requiresApproval"/>
            <parameter name="effectEnumId"><description>Optional Curator override; defaults to the verb-derived effect.</description></parameter>
            <parameter name="sourceComponent"/>
            <parameter name="statusId"/>
        </in-parameters>
        <out-parameters><parameter name="toolId"/><parameter name="toolName"/></out-parameters>
        <actions>
            <!-- 1. validate the service resolves (fail-loud; reuse ToolSchemaBuilder resolution) -->
            <script><![CDATA[
                try {
                    org.moqui.ai.ToolSchemaBuilder.build(ec.factory, serviceName)
                } catch (Exception e) {
                    ec.message.addError("Unknown or invalid service for tool: ${serviceName} (${e.message})")
                }
            ]]></script>
            <return if="ec.message.hasError()"/>

            <!-- 5a. derive toolName = verb_noun, snake_case, wire-safe -->
            <script><![CDATA[
                toolName = ((verb ?: "") + "_" + (noun ?: "")).toLowerCase().replaceAll('[^a-z0-9_]', '_')
            ]]></script>

            <!-- 5b. uniqueness: a different tool already owning this name is an error -->
            <entity-find entity-name="moqui.ai.AiTool" list="nameClash">
                <econdition field-name="toolName" from="toolName"/>
            </entity-find>
            <iterate list="nameClash" entry="other">
                <if condition="other.toolId != toolId">
                    <return error="true" message="A tool named '${toolName}' already exists (toolId ${other.toolId}); choose a different verb/noun."/>
                </if>
            </iterate>

            <!-- 2. derive effect from the verb (v1 heuristic) unless the Curator overrode it -->
            <if condition="!effectEnumId">
                <script><![CDATA[
                    def readVerbs = ['get','find','list','view','search','check','calculate'] as Set
                    effectEnumId = readVerbs.contains((verb ?: "").toLowerCase()) ? 'AI_TOOL_READ_ONLY' : 'AI_TOOL_MUTATING'
                ]]></script>
            </if>

            <!-- 4. effect-based defaults (before the denylist floor) -->
            <if condition="exposable == null">
                <set field="exposable" from="effectEnumId == 'AI_TOOL_READ_ONLY' ? 'Y' : 'N'"/></if>
            <if condition="requiresApproval == null">
                <set field="requiresApproval" from="effectEnumId == 'AI_TOOL_READ_ONLY' ? 'N' : 'Y'"/></if>

            <!-- 3. denylist floor: any matching pattern forces exposable=N, non-overridable -->
            <entity-find entity-name="moqui.ai.AiToolDenylist" list="denyList"/>
            <iterate list="denyList" entry="deny">
                <if condition="serviceName ==~ (deny.servicePattern as String)">
                    <set field="exposable" value="N"/>
                </if>
            </iterate>

            <set field="statusId" from="statusId ?: 'AI_TOOL_ACTIVE'"/>
            <!-- create vs update by stable id -->
            <if condition="!toolId">
                <set field="toolId" from="ec.entity.sequencedIdPrimary('moqui.ai.AiTool', null, null)"/>
                <set field="createdByUserId" from="ec.user.userId"/>
            </if>
            <service-call name="store#moqui.ai.AiTool" in-map="[toolId: toolId, toolName: toolName,
                verb: verb, noun: noun, description: description, serviceName: serviceName,
                effectEnumId: effectEnumId, exposable: exposable, requiresApproval: requiresApproval,
                sourceComponent: sourceComponent, createdByUserId: createdByUserId, statusId: statusId]"/>

            <!-- refresh the in-memory catalog so the new/edited tool is immediately grant-eligible -->
            <script>ec.factory.getTool("AI", org.moqui.ai.AiToolFactory.class).refreshCatalog()</script>
        </actions>
    </service>
</services>
```
> `store#moqui.ai.AiTool` (entity-auto) does create-or-update by PK; we wrap it with the gate. `createdByUserId` is set only on create (Moqui entity-auto store won't overwrite with null because we don't pass it on update — passing null would null it; on update `createdByUserId` is unset in the in-map so the existing value is preserved by store-by-pk semantics — verify and, if store nulls unspecified fields, switch to explicit create#/update# branches).

## Open questions / risks

1. **`store#moqui.ai.AiTool` entity-auto and unspecified fields (createdByUserId).** Moqui's entity-auto `store#` updates by PK; whether it nulls fields NOT present in the in-map vs. leaves them is the load-bearing assumption in Task 6 Step 3 for preserving `createdByUserId` on update. **Verify at implementation**; if `store#` nulls unspecified fields, split into explicit `create#`/`update#` branches (find-by-id first). Same caution for `store#AiAgent` (Task 7) and any field a rename-update must preserve.

2. **Catalog build timing / authz inside `DefinitionLoader.loadCatalog`.** The loader reads `AiTool` rows; it runs `disableAuthz()` defensively (reference-data read), but it's invoked lazily from request context (first `getToolByName`) — confirm reading `moqui.ai.AiTool` under a normal authenticated run doesn't need the disable (the `AiSecurityData` grant covers `moqui.ai.*` for `ALL_USERS`). If it does, the disable/enable wrapper is correct; if it interferes with an outer disable, guard with the returned boolean (as written). **Verify the boot path** — `ToolFactory.init` must NOT touch the DB (no ec yet); the lazy build (reconciliation/Task 5 Step 3 option a) is the safeguard — confirm no other code calls the old eager `loadCatalog` at init.

3. **`effect` verb heuristic fidelity.** Read verbs are a fixed allowlist (`get/find/list/view/search/check/calculate`); everything else → MUTATING (spec §6.2, v1). A read-only service named with a non-listed verb (e.g. `count#`, `export#`) is mis-classified MUTATING (safe direction: defaults to not-exposable + approval). Deeper "does it write entities?" static analysis is deferred (spec open question). Curator override via `effectEnumId` param is the escape hatch. **Acceptable for v1; note it.**

4. **`run#OmsAssistant` no longer self-heals.** Deleting `ensure#OmsAssistant` means the Playground assumes the seed ran (`gradlew load`). On a DB where the NotNaked `ext-seed` was NOT loaded, the Playground errors "Unknown agent: nn-oms-assistant" instead of auto-creating. This is the intended trade (seed is the source of truth), but it changes the fresh-DB Playground UX — **flag for the demo runbook** (load data before demoing). Acceptance Task 9 Step 7 covers it.

5. **`ext-seed` not applied on upgrade (UDM §3.7).** The NotNaked agent/tool seed AND the moqui-ai `AiToolStatus`/`AiToolEffect`/denylist seed are in `ext-seed` files. Pre-release this is fine (no existing envs). **At release**, the StatusItem/Enumeration/denylist rows (which must reach existing envs) must move to `install`/`ext` data, and the NotNaked agent seed needs an `ext-upgrade` mirror — otherwise an upgraded environment boots without effect enums (catalog build + `store#AiTool` would fail). Tracked as a release-time follow-up, out of scope for this keystone.

6. **Screen `grantList` display loses the human-readable tool name.** `AiAgentTool` now stores `toolId`, not `toolName`; the Agents screen's grant list showed `toolName`. Task 9 Step 5 downgrades it to show ids (minimal). A nicer fix (join `AiAgent`/`AiTool` for names via a view entity) is deferred — the operational UI polish is its own backlog item (`docs/plans/2026-06-04-operational-ui.md`). **Flag, don't gold-plate here.**

7. **Multi-tenant catalog scoping is explicitly out of scope** (spec §2/§11). No `ownerPartyId`/tenant column is added. If added later, it attaches to `AiTool`/`AiAgent` and the catalog query in `DefinitionLoader` + the `store#` services gain a tenant condition. Noted so the keystone doesn't silently foreclose it.

8. **Concurrent catalog refresh vs. in-flight runs.** `refreshCatalog()` swaps the catalog map atomically (volatile reference), but a long run that already built `toolSchemas` keeps its snapshot — a tool disabled mid-run still dispatches for that run. Acceptable (matches the run-time-snapshot philosophy already used for agent config in `resume`); note it.
